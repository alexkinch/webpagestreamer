# A/V drift fix via WebRTC ingest — design

**Date:** 2026-04-17
**Branch:** `fix/av-drift`
**Status:** approved, pending implementation plan

## Problem

Audio/video drift in the current architecture. Today the Chrome extension captures the tab via `MediaRecorder`, produces a continuous WebM byte stream, and ships it over WebSocket to `relay/server.js`, which pipes it into FFmpeg's stdin. FFmpeg transcodes to MPEG-TS.

Drift originates at the transport: `MediaRecorder` batches audio and video into 20 ms WebM clusters carried over a single byte stream. By the time FFmpeg sees the data, independent A/V source clocks have been collapsed onto the WebSocket arrival timeline. Previous fixes (`-async 1`, `aresample=async=1000`, `-use_wallclock_as_timestamps`) treat symptoms at the FFmpeg layer; `-use_wallclock_as_timestamps` in particular makes drift worse because it stamps every packet in a batched chunk with the same arrival time.

## Success criteria

A/V offset stays within ±40 ms (one frame at 25 fps) indefinitely, measured against a bundled test clock page.

## Non-goals

- Preserving the `rtp://`, `tcp://`, or file output schemes (breaking change; project has <10 users).
- Adding a dashboard, metrics exporter, or bitrate-adaptation UI.
- Automating the drift measurement in CI.

## Architecture

Four processes in the container, supervised by supervisord:

```
Chrome tab ──► extension (WebRTC/WHIP publisher) ──HTTP POST──► mediamtx ──RTSP──► FFmpeg ──► UDP multicast
                                                                                       └─► HTTP progressive (/stream.ts)

relay/server.js ──► IPTV endpoints (/playlist.m3u, /guide.xml, /health), test clock, FFmpeg parent
```

**Why this fixes drift:** WebRTC carries Opus and VP8 as independent RTP streams, each with its own source-clock timestamps. mediamtx preserves those timestamps through to the RTSP output. FFmpeg reads RTSP with clean PTS/DTS and no longer has to guess.

## Components

### Chrome extension

`background.js` — unchanged. Responds to `get-stream-id` via `chrome.tabCapture.getMediaStreamId()`.

`content.js` — rewritten to publish via WHIP:

1. Receive `CAPTURE_COMMAND` from the trigger (unchanged).
2. Get stream ID from background (unchanged).
3. `getUserMedia()` with tab capture constraints (unchanged).
4. Create `RTCPeerConnection`; add audio + video tracks.
5. Pin codecs to Opus + VP8 via `setCodecPreferences()`.
6. Pin video bitrate via `RTCRtpSender.setParameters({ encodings: [{ maxBitrate }] })` so localhost bandwidth estimation can't degrade quality.
7. `createOffer()` → `setLocalDescription()` → POST SDP to `http://127.0.0.1:8889/live/whip`.
8. Apply answer SDP.
9. On `iceconnectionstatechange` → `failed`/`disconnected`: tear down, wait 2 s, restart from step 2.

`forceFrames()` and the scrollbar/overlay-hiding styles stay.

`manifest.json` — add `http://127.0.0.1:8889/*` to `host_permissions` for the WHIP POST.

`trigger-capture.sh` — unchanged.

### mediamtx

New process, priority 15. Single static binary added to the Alpine image (~25 MB).

Config (`/etc/mediamtx.yml`, generated by `start.sh`):

```yaml
logLevel: info
logDestinations: [stdout]

rtsp: yes
rtspAddress: :8554
webrtc: yes
webrtcAddress: :8889
webrtcLocalUDPAddress: :8189
hls: no
rtmp: no
srt: no
api: no

paths:
  live:
    source: publisher
```

No authentication — all mediamtx ports bind to localhost only. `runOnReady` is intentionally not used; relay owns FFmpeg lifecycle.

### FFmpeg consumer

Spawned by relay. Reads RTSP, transcodes, emits MPEG-TS on stdout:

```
ffmpeg -fflags +genpts -rtsp_transport tcp -i rtsp://127.0.0.1:8554/live \
  -c:v $VIDEO_CODEC -s ${WIDTH}x${HEIGHT} -r $FRAMERATE \
  -b:v $VIDEO_BITRATE -maxrate $VIDEO_BITRATE -bufsize 2000k \
  -pix_fmt yuv420p -g $GOP -bf 2 [-flags +ilme+ildct if interlaced] \
  -vf setsar=$SAR \
  -c:a $AUDIO_CODEC -b:a $AUDIO_BITRATE -ar 48000 -ac 2 \
  -fps_mode cfr \
  -f mpegts pipe:1
```

Removed: `-use_wallclock_as_timestamps`, `aresample=async=1000`, `-async`. RTP timestamps are authoritative.

### Relay (`relay/server.js`)

Scope shrinks ~60%. Responsibilities:

- HTTP server on `HTTP_PORT` (default 9000):
  - `GET /playlist.m3u` — unchanged.
  - `GET /guide.xml` — unchanged.
  - `GET /health` — checks mediamtx RTSP reachable + FFmpeg alive + at least one output sink active.
  - `GET /stream.ts` — HTTP progressive MPEG-TS fan-out (when enabled).
  - `GET /test/clock.html` — bundled test clock (see below).
- Spawn FFmpeg; restart on exit with 2 s backoff.
- Fan out FFmpeg stdout to configured sinks:
  - UDP multicast (dgram, 1316-byte TS-aligned packets).
  - HTTP progressive clients (existing `httpStreamClients` Set pattern).

Deleted: WebSocket server, WebM stdin path, `outputHandler` abstraction, `rtp://`/`tcp://`/file handlers.

### Test clock

`test/clock.html` served at `GET /test/clock.html`. Renders:

- Large UTC time to millisecond precision, updated via `requestAnimationFrame`.
- WebAudio oscillator emitting a 50 ms tone at each second boundary.

Used for drift measurement and as the default `URL` in dev builds so the test loop is self-contained.

## Data flow

```
tabCapture stream → RTCPeerConnection (VP8/Opus RTP)
                   → WHIP POST to mediamtx:8889
                   → mediamtx buffers, maintains independent RTP timestamps
                   → RTSP at mediamtx:8554/live
                   → FFmpeg (transcode to profile)
                   → MPEG-TS on stdout
                   → relay fan-out
                      ├─ UDP multicast (if UDP_OUTPUT set)
                      └─ HTTP progressive (if HTTP_OUTPUT=true)
```

## Environment variables

Breaking changes:

| Old | New | Notes |
|---|---|---|
| `WS_PORT` | `HTTP_PORT` | WebSocket gone; `WS_PORT` kept as silent fallback for one release. |
| `OUTPUT=udp://...` | `UDP_OUTPUT=udp://...` | Split so UDP and HTTP can run concurrently. |
| `OUTPUT=http` | `HTTP_OUTPUT=true` | Same. |
| `OUTPUT=rtp://...`/`tcp://...`/file path | — | Removed. |

Unchanged: `URL`, `WIDTH`, `HEIGHT`, `FRAMERATE`, `PROFILE`, `VIDEO_CODEC`, `AUDIO_CODEC`, `VIDEO_BITRATE`, `AUDIO_BITRATE`, `SAR`, `INTERLACED`, `CHANNEL_NAME`, `CHANNEL_ID`, `PROGRAMME_TITLE`, `PROGRAMME_DESC`, `STREAM_URL`, `CDP_PORT`.

## Error handling

| Failure | Recovery |
|---|---|
| Chrome dies | supervisord restart → trigger re-fires → extension re-publishes → mediamtx emits fresh RTSP → relay restarts FFmpeg on RTSP read failure. |
| mediamtx dies | supervisord restart → FFmpeg RTSP read fails → relay restarts FFmpeg with 2 s backoff. Extension's `iceconnectionstatechange` handler re-publishes. |
| FFmpeg dies | relay restarts it with 2 s backoff. |
| Relay dies | supervisord restart; FFmpeg gets SIGTERM via process group; mediamtx + Chrome keep running. |

## Supervisord priorities

| Priority | Process | Autorestart |
|---|---|---|
| 10 | relay | yes |
| 15 | mediamtx | yes |
| 20 | chrome | yes |
| 30 | trigger | on unexpected exit only |

Relay comes up first so the health endpoint is available during bring-up.

## Testing

1. **Build + smoke.** `docker build`, run with defaults, consume UDP with `ffplay`, visually verify playback.
2. **Drift measurement.** Run container pointed at bundled `/test/clock.html`. Record 60 min of the UDP stream: `ffmpeg -i udp://@239.0.0.1:1234 -t 3600 -c copy drift-test.ts`. Spot-check video-frame timestamp vs audio-beep envelope at t=0, t=30 min, t=60 min. Pass if all three offsets are within ±40 ms.
3. **Restart resilience.** Kill Chrome mid-stream; confirm UDP output resumes within 10 s with sync intact.
4. **CI.** Existing `docker build` check stays. Drift measurement is manual and documented in README.

## Rollout

Rip-and-replace on `fix/av-drift`. No side-by-side coexistence; old `relay/server.js` WebSocket + WebM path is deleted in the same change. Release notes call out env var changes and removed output schemes.
